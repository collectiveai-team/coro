# Capability-First Backend Layout

Supersedes the Provider-First Backend Layout decision in ADR 0001.

ML Model Integration modules are organized under `coro/backends/` by capability first: `coro/backends/asr/` holds the ASR adapters (faster-whisper, onnx-asr, onnx-genai) and `coro/backends/diarization/` holds the diarization adapters (NeMo batch Sortformer, NeMo streaming Sortformer, pyannote). Each capability subpackage owns a Backend Adapter Factory module that dispatches on the configured Backend Provider: `coro/backends/asr/factory.py` builds an ASR Adapter and `coro/backends/diarization/factory.py` builds a Diarization Adapter, reports whether a provider supports streaming, and constructs the streaming diarizer for streaming-capable providers. The application factory depends only on these factories and the core protocols; it imports no specific provider and reaches into no adapter internals.

This reverses ADR 0001, which chose a provider-first layout so a single provider module could own both ASR and diarization integration. That rationale assumed the dominant axis of change is "the provider"; in practice the dominant axis is "the capability". Pipelines consume an ASR Adapter and a Diarization Adapter as separate capabilities (CONTEXT.md), the Backend Adapter Factory is naturally per-capability (one factory builds ASR Adapters, another builds Diarization Adapters), Server Startup Selection chooses an ASR Backend Provider and a Diarization Backend Provider independently, and provider growth has been capability-scoped (a new diarization engine, pyannote, alongside multiple ASR-only providers). Grouping by capability puts the modules that share a protocol, a factory, and a startup selector next to each other, and makes "add a diarization engine" a localized change in one subpackage.

The accepted trade-off is that NeMo supplies diarization in two forms (batch Sortformer and streaming Sortformer) but no ASR adapter today, so NeMo lives wholly under `coro/backends/diarization/` as a `nemo/` provider subpackage rather than spanning two capability trees. If a future provider exposes both ASR and diarization from one engine, its integration is split across `backends/asr/` and `backends/diarization/`, with any genuinely shared, backend-specific setup extracted into a small shared module imported by both — not duplicated. Cross-capability native-output normalization that is not backend-specific (e.g. speaker-label and segment coercion) lives in a shared diarization module (`coro/backends/diarization/segments.py`) consumed by every Diarization Adapter, so NeMo and pyannote emit an identical speaker timeline shape without duplicated normalization.

Protocols stay in the Core Boundary unchanged: `ASRAdapter`, `DiarizationAdapter`, `StreamingDiarizerFactory`, and `TranscriptionPipeline` remain under `coro.core`, and only concrete adapters live under `coro.backends`. The concrete NeMo streaming-diarizer factory is named `NemoStreamingDiarizerFactory` so it no longer collides by name with the `StreamingDiarizerFactory` protocol it implements.

Considered alternatives included keeping the Provider-First Backend Layout from ADR 0001 (rejected: it splits the per-capability factory and forces the application factory to know providers; it optimizes for an axis of change that has not materialized), a flat `backends/` of provider modules with no subpackages (rejected: the current flat tree already mixes ASR and diarization modules with no grouping, and a per-capability factory has no natural home), and a hybrid where ASR stays flat while only diarization gets a subpackage (rejected: asymmetric, leaves ASR construction branching inline in the application factory, and obscures the parallel between the two capabilities). The chosen shape keeps each capability's adapters, factory, and shared helpers co-located, keeps the application factory backend-agnostic, and preserves core-owned protocols.
