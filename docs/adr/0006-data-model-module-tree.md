# Data Model Module Tree

Data models are defined in dedicated, concern-grouped model modules rather than inline next to behavior. Project-Owned Transcript Model types live under `coro/core/models/` (`transcript.py` for tokens and segments, `response.py` for the Transcription Response Model, `events.py` for pipeline stream events). The previous monolithic `coro/core/types.py` is removed; its contents moved into this tree and all importers use `coro.core.models`. No back-compat re-export shim is kept.

The split policy is: a model that crosses a module boundary â€” produced in one module and consumed in another, or asserted on by tests across module boundaries â€” moves out of the behavior module into a model module. A model that is a truly module-private one-off â€” only constructed and consumed inside its own module and never imported across a boundary â€” may stay in-file.

Applying the rule, the following private one-offs intentionally stay local: `LatencyTierParams` in `coro/backends/nemo_streaming.py` (backend-specific streaming parameters, never imported elsewhere; see ADR 0005), the `/proc` parsing structs `SmapsRollup`, `ProcIo`, `ProcStat`, and `ProcStatus` in `coro/bench/run.py` (private read details), and the raw NVML `GpuDevice` in `coro/bench/gpu.py` (a private device-read detail; only the process-attributed `GpuSample` crosses out).

Benchmark tooling models live under `coro/bench/models/` (`resource.py`, `gpu.py`, `performance.py`, `quality.py`, `report.py`), not under `coro/core/`. Bench is internal tooling, not a domain boundary, and the Core Boundary is reserved for API-agnostic domain models, interfaces, and pure transformations. Coupling bench shapes into `core/` purely for tidiness would invert the dependency direction (`bench` depends on `core`/`api`, not the reverse), so bench keeps its own model tree. The Pydantic Boundary Response Schema continues to live at the API boundary in `coro/api/schemas.py` per ADR 0001; only the internal response dataclasses were relocated.

Model relocations preserve dataclass field order, defaults, and `init=False` sentinels verbatim, because `dataclasses.asdict` output is load-bearing: the SSE done frame is byte-identical to the materialized response (`tests/test_done_frame.py`), and bench on-disk artifact shapes are read back by field name. Moving a definition must not change its serialized shape.

Considered alternatives included keeping `coro/core/types.py` as a permanent re-export shim (rejected: leaves a rotting indirection layer), placing bench models under `core/models/bench/` for a single model root (rejected: couples internal tooling into the domain Core Boundary), and relocating every dataclass including private parsing structs (rejected: pollutes model modules with backend and `/proc`-read internals that no other module consumes). The chosen shape keeps `core` focused on domain boundaries, keeps behavior modules free of in-file model clutter, respects the established layering, and leaves a documented, principled exception for module-private one-offs.
